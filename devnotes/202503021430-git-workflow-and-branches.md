# Git・SourceTree 運用ルール（2台開発用）

- **最終更新**: 2025年3月2日

## 本機・ノート機でのブランチの扱い

- **本機**: **develop** ブランチで作業する。DB・メディアの主環境であり、develop を統合ブランチとして維持する。
- **ノート機**: **develop を pull した上で、作業用に `feature/xxx` を切って作業する**ことを推奨。
  - 作業完了後、そのブランチを **develop にマージ**（本機でマージしてもノート機でマージしても可）してから push。
  - 本機で「取得」→ develop を pull すれば、ノート機でマージした内容が取り込まれる。
- **ノート機で develop のまま作業する**ことも可能（小さい修正など）。その場合は「同じファイルを本機と同時に編集しない」ことに注意。並行して機能単位で作業する場合は、ノート機は feature ブランチを使うとコンフリクトを避けやすい。

## ブランチ戦略

| ブランチ | 用途 |
|---------|------|
| `main` | 本番反映済み・安定版。直接コミットは避け、PR/マージで更新。 |
| `develop` | 開発統合用。日々の作業はここにマージ。 |
| `feature/xxx` | 機能・ページ単位の作業（例: `feature/contact-form`）。 |
| `fix/xxx` | バグ修正・軽微な修正。 |

- 作業開始時: `develop` から `feature/xxx` を切る（または `develop` で作業）。
- 作業完了後: `feature/xxx` を `develop` にマージ。本番反映時は `develop` を `main` にマージ。

## 2台での基本ルール

1. **作業開始前**: 必ず `git pull`（または SourceTree で「プル」）してから編集する。
2. **コミット**: 意味のまとまりでコミット。メッセージは日本語可（例: `[フェーズ0] .gitignore 追加`）。
3. **push 前**: 再度 `git pull` してから `git push`。コンフリクトはその場で解消。
4. **コンフリクト時**: 片方で解消してコミット → もう片方でプルしてから作業再開。

## SourceTree での推奨手順

- **本機**: リモートの「取得」→ develop を最新化 → develop で作業 → コミット → プッシュ。
- **ノート機（feature で作業する場合）**: 「取得」→ develop を pull → develop から `feature/xxx` を新規作成 → そのブランチで作業 → コミット → プッシュ（feature ブランチを push）→ 完了したら develop にチェックアウトして feature をマージ → develop をプッシュ。
- **ノート機（develop のまま作業する場合）**: 「取得」→ develop を pull → develop のまま作業 → コミット → プッシュ。

## コミットメッセージ例

- `[フェーズ0] docker-compose と .gitignore を追加`
- `[フェーズ5] トップページテンプレート追加`
- `fix: 問い合わせフォームのバリデーション修正`

以上を devnotes に残し、両マシンで同じルールで運用する。
